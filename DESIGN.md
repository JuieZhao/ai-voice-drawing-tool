# 设计文档

## 1. 需求理解

比赛议题要求开发一款纯语音控制的绘图工具。用户不能使用鼠标或键盘完成绘图创作，只能通过语音指令驱动画布。

核心评价点包括：

1. 指令理解的准确性与容错性。
2. 语音到绘图操作的响应延迟。
3. 复杂指令的拆解与执行能力。
4. 作品完整度、交互流畅度和创新性。

## 2. 产品定位

本项目定位为一个语音驱动的矢量绘图白板。

它不把语音直接转换成不可编辑图片，而是把语音转换为可执行的绘图 DSL，再由 Canvas 渲染为可继续编辑的图形对象。

## 3. 用户故事

计划支持：

| 用户故事 | 当前状态 |
| --- | --- |
| 用户可以启动语音监听 | 已实现 |
| 用户可以语音创建基础图形 | 已实现 |
| 用户可以指定颜色、位置和大小 | 已实现 |
| 用户可以使用 A1-C3 坐标格定位对象 | 已实现 |
| 用户可以修改最近创建对象 | 已实现 |
| 用户可以撤销、重做、清空画布 | 已实现 |
| 用户可以一笔一笔画路径 | 已实现初版 |
| 用户可以一句话生成多个对象 | 部分实现 |
| 用户可以通过 DeepSeek 解析复杂指令 | 可选实现 |
| 用户可以查看路径动作的绘制步骤 | 已实现初版 |

| 用户可以像海龟绘图一样逐步控制画笔 | 已实现初版 |

## 4. 系统架构

```text
浏览器前端
  ├─ Web Speech API 语音识别
  ├─ 本地规则指令解析器
  ├─ 可选 DeepSeek 指令解析器
  ├─ 绘图 DSL
  ├─ 路径笔画解析器
  ├─ 海龟式画笔状态
  ├─ Canvas 渲染器
  ├─ 对象状态管理
  ├─ 撤销 / 重做栈
  └─ 执行日志

Node 本地服务
  ├─ 静态资源服务
  ├─ /api/llm-status
  └─ /api/llm-command -> DeepSeek OpenAI-compatible Chat API
```

## 5. 绘图 DSL

项目使用结构化 JSON 表达绘图动作。

示例：

```json
{
  "actions": [
    {
      "type": "create_shape",
      "shape": "circle",
      "fill": "#60A5FA",
      "position": "center",
      "size": 120,
      "style": "cute_flat"
    }
  ]
}
```

当前支持的动作：

- `create_shape`
- `draw_path`
- `update_object`
- `resize_object`
- `move_object`
- `delete_object`
- `undo`
- `redo`
- `clear_canvas`
- `pen_down`
- `pen_up`
- `turtle_forward`
- `turtle_turn`
- `turtle_home`
- `turtle_color`
- `turtle_width`
- `set_grid`

路径型 DSL 可以额外带 `plan` 字段，用于解释画笔动作的拆解过程：

```json
{
  "plan": ["向右画一条直线", "从上一笔末端向下画曲线", "在末端画圆"],
  "actions": [
    {"type": "draw_path", "path": "line", "direction": "right", "distance": 120, "anchor": "cursor"},
    {"type": "draw_path", "path": "curve", "direction": "down", "distance": 100, "anchor": "last_end"},
    {"type": "draw_path", "path": "circle", "radius": 42, "anchor": "last_end"}
  ]
}
```

## 6. 指令理解策略

当前 MVP 采用“本地规则优先 + DeepSeek 可选增强”的混合解析策略，优先保证低延迟、可控性和 Demo 稳定性。

解析流程：

```text
中文语音文本
  -> 清理空格和标点
  -> 判断系统命令
  -> 拆分多段指令
  -> 识别颜色、位置、大小、坐标格、路径类型和上下文锚点
  -> 生成绘图 DSL
  -> 执行动作
```

复杂口令会通过本地 Node 服务调用 DeepSeek，让模型输出 JSON；如果未配置 Key、网络失败或模型返回内容不可执行，则自动回退到本地规则解析。

对于“画一只小猫”“画一辆小汽车”这类完整物体，当前不再用预置对象兜底，而是提示用户先用“画直线、画曲线、画圆、从上一笔继续”描述步骤。这样可以先把真实绘图过程和上下文打牢，再逐步扩展到 LLM 生成可执行路径。

为降低语音描述位置的难度，画布默认显示 A1-C3 九宫格辅助线。用户既可以说“左上角、右边、下面”等自然方位，也可以说“在 B2 画一个圆”“把它移到 C3”。坐标格只作为构图辅助，不改变画布对象模型；画面上不直接显示 A1/B2 标签，避免干扰作品观感。

画布上的细网格同时承担距离单位：1 格 = 34 像素。用户说“向右画一个五格长的直线”时，解析器会生成 `gridUnits: 5`，执行层再换算成实际像素距离。这样用户不需要理解像素，也能像写 turtle 代码一样精确控制笔画长度。

## 7. Prompt 设计

DeepSeek 系统 Prompt 采用以下原则：

```text
你是一个语音绘图指令解析器。你的任务是把用户的中文语音指令转换为绘图操作 JSON。

规则：
1. 只能输出 JSON，不要输出解释。
2. 如果一句话包含多个绘图动作，请拆分成 actions 数组。
3. 优先使用系统支持的基础图形、路径动作和样式参数。
4. 如果用户说“它”“刚才那个”“这个”，优先引用最近创建或最近选中的对象。
5. 如果目标对象无法确定，返回 clarification 字段，向用户提出一个简短问题。
6. 不要输出贴图或组合模板；复杂对象也必须拆成路径动作和基础图形。
7. 不要输出无法执行的自由描述。
8. 所有图形默认使用 cute_flat 风格。
```

实现约束：

1. API Key 只保存在本地服务端环境变量或 `.env`，不会暴露给浏览器。
2. DeepSeek 输出必须经过前端 DSL 白名单过滤，只允许执行已支持动作。
3. 默认模型为 `deepseek-v4-flash`，如需更强理解可配置为 `deepseek-v4-pro`。
4. 默认关闭 DeepSeek thinking mode，降低延迟并减少 JSON 输出不稳定性。

## 8. 审美系统

项目采用儿童绘本式扁平矢量贴纸风。

视觉规则：

- 深色统一描边
- 明亮柔和配色
- 圆角和柔和曲线
- 少量阴影
- 不追求真实照片感，追求清晰、可爱、可编辑

## 9. 路径与上下文策略

当前版本主动撤掉组合模板，避免作品变成“语音贴纸工具”。系统把每一笔都记录为可编辑对象，并为它维护中心、边界和末端等上下文锚点。

系统采用 turtle-like 的思想，但实现上不直接依赖 Python turtle。核心链路是：

```text
自然语言 -> 绘图 DSL -> Canvas 路径对象 -> 可继续引用的上下文
```

关键上下文包括：

- 当前画笔位置和方向。
- 最近一笔的末端。
- 最近对象的中心、左侧、右侧、上侧、下侧。
- 最近对象引用：“它”“刚才那个”“上一笔”。

典型链式口令：

```text
向右画一条直线
向右画一个五格长的直线
接着向下画一条曲线
在末端画一个圆
从刚才的圆右边继续画直线
```

## 10. 海龟式画笔

为了解决“只会放库里的形状”的局限，项目新增了海龟式画笔状态：

- 画笔位置：使用画布归一化坐标保存。
- 画笔方向：0 度向右，右转为正角度。
- 落笔状态：落笔时移动会生成线条，抬笔时只移动光标。
- 画笔属性：支持颜色和线宽。
- 路径预设：支持“用画笔画正方形 / 三角形”，自动拆成落笔、前进、转向、抬笔动作。

典型口令：

```text
落笔
向前走一百
向右转九十度
再向前走六十
抬笔
回到中心
用画笔画一个正方形
用画笔画一个三角形
```

这个能力让用户可以逐步“教”系统画路径，也为 LLM 把复杂轮廓拆成路径动作提供基础。

## 11. 延迟优化

当前版本使用本地规则解析，不依赖网络模型，因此常用指令响应很快。

接入 LLM 后使用混合策略：

1. 简单命令本地解析。
2. 复杂指令调用 DeepSeek。
3. LLM 输出必须经过 DSL 白名单过滤。
4. 解析失败时降级为澄清问题或简化版本。

## 12. 未完成部分与原因

| 功能 | 原因 |
| --- | --- |
| JSON Schema 校验 | 当前先用白名单过滤，后续再抽成显式 Schema |
| PNG / SVG 导出 | MVP 优先验证语音绘图闭环 |
| 更丰富路径类型 | 贝塞尔曲线、圆弧和闭合路径仍需继续扩展 |
| 复杂物体自动规划 | 需要 LLM 输出稳定路径步骤，而不是返回贴图模板 |
| 语音澄清对话 | 需要更完整的对话状态管理 |
| 更强上下文选择 | 需要区分“上一条线”“左边的圆”“最近的曲线”等目标 |

## 13. 后续优化方向

1. 接入 LLM 解析复杂自然语言。
2. 扩展 `draw_path`，支持贝塞尔曲线、圆弧和路径闭合。
3. 支持导出作品。
4. 支持语音确认和语音澄清。
5. 增加对象选择能力，例如“选择上一条曲线”。
6. 增加自动构图，使复杂路径组合更美观。
7. 让 LLM 输出“绘制步骤 + 路径动作 + 基础图形”，避免硬编码模板依赖。
